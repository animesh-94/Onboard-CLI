Before Onboard-CLI can analyze, visualize, or lint your architecture, it must build a local intelligence graph. The `init` command handles the scaffolding of the `.onboard` directory and sets up the base `onboard.yaml` and `architecture.yaml` configuration files for your project.

## Usage

```bash
# Normal Initialization
onboard init

# CI/CD Initialization with Specific Template
onboard init --template clean-architecture
```

### Options

- `--template [name]`: Specifies the `architecture.yaml` template to scaffold. Available templates are `generic` (default), `clean-architecture`, `modular-monolith`, `mvc`, and `serverless`.

## What happens under the hood?

When you run `onboard init`, the CLI performs the following actions:

1. **Scaffolds Directories**: It creates a hidden `.onboard` directory in the root of your project. This directory will act as the home for your local SQLite graph cache, topology dumps, and rulesets.
2. **Generates Configs**: It drops a base `onboard.yaml` file (for CLI config like ignored dirs) and an `architecture.yaml` file (pre-populated based on the selected template) into your project.
3. **Prepares for Mapping**: The `.onboard/cache.db` file is provisioned and the SQLite tables and indexes are initialized, preparing the engine for the `map`, `drift`, and `export` commands.
