Documentation decays; code is reality. `onboard drift` uses the static AST graph to find architectural violations that standard text-based linters miss, ensuring your intended system design matches the actual execution path.

## Usage

```bash
onboard drift --ruleset <path_to_yaml>
```

### The Ruleset File (architecture.yaml)
To enforce boundaries, you must define them. Create a YAML file in your repository root defining your system's layers and the import restrictions between them.

```yaml
# 1. Define your architectural boundaries (using regex paths)
boundaries:
  api_layer: "src/api/.*"
  domain_layer: "src/domain/.*"
  database_layer: "src/db/.*"

# 2. Define the strict rules
rules:
  - source: api_layer
    cannot_import: database_layer
    message: "Controllers must not talk directly to the DB. Route through the domain_layer."
  
  - source: domain_layer
    cannot_import: api_layer
    message: "Domain logic must be framework-agnostic. Do not import HTTP handlers."
```

### Output & CI/CD Integration
When you run `onboard drift`, the CLI traverses the AST and cross-references all cross-file imports against your YAML ruleset.

```bash
onboard drift --ruleset architecture.yaml
```

**Terminal Output:**
```plaintext
Analyzing drift using ruleset: architecture.yaml

❌ Architectural Violations Detected:
 - [Line 42] src/api/order_controller.go: api_layer cannot import database_layer (Rule broken: Controllers must not talk directly to the DB. Route through the domain_layer.)
 - [Line 18] src/domain/user_service.go: domain_layer cannot import api_layer (Rule broken: Domain logic must be framework-agnostic. Do not import HTTP handlers.)
exit status 1
```

### Blocking PRs in GitHub Actions
Because `onboard drift` returns a non-zero exit code (`exit 1`) when a violation is found, it is the perfect tool to run in your CI pipeline to prevent architectural degradation.

We have provided a ready-to-use GitHub Actions workflow template at [`docs/onboard-action.yml`](https://github.com/animesh-94/Onboard-CLI/blob/main/docs/onboard-action.yml). You can simply copy this into your `.github/workflows/` directory to instantly enforce boundaries on every Pull Request.

```yaml
name: Architecture Check
on: [pull_request]

jobs:
  lint-architecture:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.22'
      - name: Install Onboard CLI
        run: go install github.com/animesh-94/Onboard-CLI@latest
      - name: Run Drift Check
        run: onboard drift --ruleset architecture.yaml
```
